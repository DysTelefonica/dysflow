# Design — Unify MCP tool registration

- Change: `405-unify-mcp-tool-registration`
- Scope: `src/adapters/mcp/tools.ts` (+ tests under `test/adapters/mcp/`)

## Current state (what we are replacing)

`registerMcpTools(currentTools, ...)`:

1. Seeds `tools` with the 5 modern tools (`currentTools`).
2. Adds 10 alias tools via `add({...})` (explicit inline handlers).
3. Loops `for (const toolName of DYSFLOW_MCP_TOOL_NAMES) add(createDispatchTool(toolName, ...))`.
4. `add` is dedup-by-`Set`: `if (!names.has(tool.name)) { names.add; tools.push }` — **silently
   skips** duplicates.

The 10 alias names are a subset of `DYSFLOW_MCP_TOOL_NAMES`, so step 3 re-emits them and step 2's
entries win only because they were pushed first. The collision is invisible.

## Chosen design — one registration list, one pass, hard duplicate check

Keep the same two *kinds* of handler (bespoke alias handlers vs. generated dispatch handlers) but
make registration **single-pass and order-independent for correctness**, and make the per-name
ownership explicit.

### 1. A name-keyed map of alias entries (the explicit overrides)

Introduce a single local structure that maps each alias name to its bespoke entry builder:

```ts
// Names that have a bespoke (alias) handler instead of the generic dispatch handler.
// This is the SINGLE place that says "these names are NOT generated by the dispatch loop".
const ALIAS_TOOL_NAMES = new Set<DysflowMcpToolName>([
  "list_access_operations",
  "cleanup_access_operation",
  "run_vba",
  "query_sql",
  "exec_sql",
  "run_script",
  "create_table",
  "drop_table",
  "seed_fixture",
  "teardown_fixture",
]);
```

The ten bespoke entries are built (as today) but collected into an ordered list, e.g.
`const aliasTools: DysflowMcpTool[] = [ ...the ten add({...}) bodies turned into array elements... ]`.
Their handler bodies are copied verbatim — no behavior change.

### 2. The dispatch loop SKIPS alias names

```ts
const dispatchTools = DYSFLOW_MCP_TOOL_NAMES
  .filter((name) => !ALIAS_TOOL_NAMES.has(name))
  .map((name) => createDispatchTool(name, services, writesEnabled, writeAccessResolver, env));
```

This is the key structural change: a given `DysflowMcpToolName` is owned by **either** the alias map
**or** the dispatch loop, never both. The overlap is gone by construction, not by insertion order.

### 3. One registration pass with a HARD duplicate check

```ts
function registerMcpTools(currentTools, ...): DysflowMcpTool[] {
  const tools: DysflowMcpTool[] = [];
  const names = new Set<string>();
  const add = (tool: DysflowMcpTool): void => {
    if (names.has(tool.name)) {
      throw new Error(`Duplicate MCP tool registration: ${tool.name}`);
    }
    names.add(tool.name);
    tools.push(tool);
  };

  for (const tool of currentTools) add(tool);   // 5 modern
  for (const tool of aliasTools) add(tool);      // 10 alias (explicit overrides)
  for (const tool of dispatchTools) add(tool);   // 35 generated dispatch (45 - 10 overridden)

  return tools;
}
```

Now the registration order still produces the same array order as today (modern, then the ten
aliases, then the remaining dispatch names in `DYSFLOW_MCP_TOOL_NAMES` order minus the aliases). Any
accidental duplicate throws instead of being absorbed.

### Why this preserves behavior exactly

- The 5 modern tools: unchanged (`currentTools` is built identically).
- The 10 alias tools: same handler bodies, registered exactly once (as before, since the alias path
  won the collision). Today the dispatch versions of these 10 were discarded; now they are never
  built — same observable result.
- The 35 remaining dispatch tools: built by the same `createDispatchTool`, in the same relative
  order (the `filter` preserves `DYSFLOW_MCP_TOOL_NAMES` order; the removed 10 were the ones the
  alias path already shadowed).
- Counts: 5 modern + 10 alias + 35 dispatch = 50 entries; 2 of the dispatch entries are hidden stubs
  (`verify_binary`, `reconcile_binary`) → 48 visible. The release-matrix gate computes
  `modernCount` as non-`DYSFLOW_MCP_TOOL_NAMES` names = 5, and `visibleCount` = 48. Both unchanged.
  (`toolCount = DYSFLOW_MCP_TOOL_NAMES.length = 45`, `stubCount = 2` are constants on the registries,
  also unchanged.)

### Ordering note (refactor-safety)

The final tool array order is preserved, but **no test should assert on order**. The behavioral
guarantee is "these names resolve to these handlers" and "these counts hold" — both order-independent
(`tools.find(t => t.name === ...)`). The existing suite already follows this. The new duplicate test
asserts a thrown error, not order.

## The duplicate-throws behavior — how to test it at the port

We cannot trigger a real duplicate through the public `createDysflowMcpTools` factory once the loop
skips aliases (that is the point). To prove the hard check is wired in without coupling to internals,
extract the registration core into a small **exported, pure** function that takes a list of tool
entries and returns the registered list, throwing on duplicate names:

```ts
export function registerMcpToolList(entries: readonly DysflowMcpTool[]): DysflowMcpTool[] {
  const names = new Set<string>();
  const out: DysflowMcpTool[] = [];
  for (const tool of entries) {
    if (names.has(tool.name)) {
      throw new Error(`Duplicate MCP tool registration: ${tool.name}`);
    }
    names.add(tool.name);
    out.push(tool);
  }
  return out;
}
```

`registerMcpTools` then calls `registerMcpToolList([...currentTools, ...aliasTools, ...dispatchTools])`.

This is a **pure function** (input list → output list / throw) — testing it is behavior testing, not
implementation coupling. The test feeds two entries with the same `name` and asserts it throws; it
never inspects private collaborators or call order. This satisfies the testing philosophy
("unit test of a pure function tests behavior").

Rationale for exporting: it gives a real port (a pure, named contract) for the
"duplicate must throw" guarantee, instead of forcing the test to monkey-patch internals or rely on a
contrived registry mutation. The export is small and intention-revealing.

## Alternatives considered

1. **Keep the `Set` dedup but log a warning on collision.**
   Rejected: still two sources of truth, still silent-ish, does not make a duplicate a hard failure,
   and a reader still must read both paths plus ordering.

2. **Fold everything into one big `entries` map keyed by name (modern + alias + dispatch) and a
   single builder.**
   Rejected for this change as over-reach: the five modern tools have genuinely bespoke
   shapes/handlers and live in `currentTools` for good reason; collapsing them buys little legibility
   and risks behavior drift. The minimal "alias map + dispatch-skip + hard add" gives single
   ownership per name with far less churn. Can be revisited later if desired.

3. **Remove the alias handlers entirely and route those ten names through the dispatch loop.**
   Rejected: the alias handlers have behavior the dispatch path does not (e.g. `run_vba` parses
   `argsJson`, builds `{ moduleName: "", procedureName, arguments }`; `query_sql` maps
   `query`→`sql` and `sourcePath`→`databasePath`; `cleanup_access_operation` defaults `accessPath`
   to `""` and goes through `cleanupService`, not the dispatch routes). Routing them through dispatch
   would CHANGE observable behavior and break tests. Behavior must be preserved, so aliases stay as
   explicit overrides.

## Risks & mitigations

- **Risk:** subtly reordering dispatch tools breaks an order-coupled assertion.
  *Mitigation:* none expected — the suite uses `tools.find(...)`; the `filter` preserves order
  anyway. If any order coupling surfaces, fix the test (philosophy: order is implementation).
- **Risk:** the alias/dispatch split list drifts from `DYSFLOW_MCP_TOOL_NAMES`.
  *Mitigation:* `ALIAS_TOOL_NAMES` is typed `Set<DysflowMcpToolName>`, so a stale name fails `tsc`.
  The release-matrix gate count test catches drift in totals.
- **Risk:** the extracted `registerMcpToolList` export is treated as public API.
  *Mitigation:* document it as an internal-but-exported helper for the registration contract; it has
  no MCP-wire surface.

## Spec delta

No observable-contract change. Tool names, schemas, descriptions, and handler outputs are identical.
**No spec delta is required.** This is an internal adapter refactor plus a hardened invariant
(duplicate-throws) and its regression guards.
