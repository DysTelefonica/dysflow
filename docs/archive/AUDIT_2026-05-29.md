# Dysflow — Audit Report v0.9.20 (2026-05-29)

## Summary

v0.9.20 closes out the full P1–P8 improvement plan from 2026-05-28. All eight items — Biome lint escalation, branch coverage increase, VBA adapter split, schema file split, SHA-256 checksum verification, HTTP Bearer token auth, MCP protocol version documentation, and `install.ts` refactor — are shipped and verified. The TypeScript layer is now structurally sound: no god objects above 510 lines, no explicit `any`, no non-null assertions, 82%+ coverage on every metric. The remaining risks are concentrated in two areas that P1–P8 did not touch: the PowerShell scripts (both still monolithic, no Pester unit coverage for the post-N2 functions), and a small set of logic gaps in the new sub-adapters (most notably `VbaOperationsAdapter.execute()` returning `TOOL_NOT_IMPLEMENTED` unconditionally for `list_access_operations` and `cleanup_access_operation` when called through the `vba-sync` MCP route, overriding the working legacy alias handlers).

---

## Strengths

- **Lint gates are now hard failures.** `noExplicitAny` and `noNonNullAssertion` are both `error` in `biome.json` (P1). The codebase has zero lint violations — confirmed by the three `as unknown as` casts in `vba-source-comparison.ts` being the only surviving unsafe casts, all intentional type-narrowing workarounds for variance limitations.
- **Coverage thresholds are symmetric.** All four metrics are now at or above 82 % (`statements: 82`, `branches: 82`, `functions: 85`, `lines: 84`). The branch threshold was the lagging metric; P2 closed the gap.
- **Supply-chain attack surface is meaningfully reduced.** `dysflow update` now verifies SHA-256 against a `SHA256SUMS` file, with a `--skip-checksum` escape hatch. The mismatch path throws with a clear message and blocks install. The happy path and mismatch path both have unit tests (`test/cli/install.test.ts:1305–1367`).
- **HTTP adapter has optional Bearer token auth** — controlled by `httpToken` in `.dysflow/project.json` or `DYSFLOW_HTTP_TOKEN` env var (`src/adapters/http/server.ts:78–135`). The `/health` endpoint is exempted from auth, which is appropriate. The auth check correctly uses constant-time string comparison via `===` (acceptable for tokens of this threat model).
- **`install.ts` is now a 144-line entry point** with clear re-exports. Six focused sub-modules under `src/cli/commands/install/` each do one thing. All are well within the 260-line ceiling.
- **MCP protocol version is named.** `MCP_PROTOCOL_VERSION = "2024-11-05" as const` is exported from `stdio.ts:37` with an upgrade comment pointing to `spec.modelcontextprotocol.io`. No more magic string.
- **Schema files are well-structured.** `schemas.ts` is now a one-line barrel (`export * from "./schemas/index.js"`). The three domain slices are 234 / 333 / 324 lines — all under the 350-line ceiling from P4.
- **P3 VBA adapter split is correct.** The orchestrator (`vba-sync-adapter.ts`, 509 lines) still handles context resolution, timeout, and preflight. Each sub-adapter is focused and under 260 lines.
- **Request body limits are enforced everywhere.** The HTTP server caps at `DEFAULT_HTTP_MAX_BODY_BYTES = 1 MiB` with a streaming counter that rejects oversized bodies mid-stream. The MCP stdio adapter caps at `DEFAULT_MAX_REQUEST_BYTES = 1 MiB` with line-level overflow detection.
- **Error sanitisation is thorough.** `sanitizeMcpErrorMessage` in `tools.ts:726–742` has five regex passes with non-catastrophic backtracking, covering UNC paths, Windows drive paths, POSIX paths, and `.accdb`/`.mdb`-specific patterns. The regex comments explain the defense strategy.

## Weaknesses

- **`VbaOperationsAdapter.execute()` is a stub that always fails.** `src/adapters/vba-sync/vba-operations-adapter.ts:37–39`: `execute(_toolName, _input)` unconditionally returns `TOOL_NOT_IMPLEMENTED` for both `list_access_operations` and `cleanup_access_operation`. These two tools have `{ kind: "vba-sync" }` in `MCP_TOOL_ROUTES` (`tools.ts:442–443`), which routes them through `services.vbaSyncToolService.execute()` → `VbaSyncAdapter.execute()` → `VbaOperationsAdapter.execute()`. The legacy-alias handlers at `tools.ts:264–301` are registered first via `add()`, so they win the `names.has()` check and the MCP dispatch never reaches the stub. But any new consumer that calls `vbaSyncToolService.execute("list_access_operations", ...)` directly will always get `TOOL_NOT_IMPLEMENTED`. The stub is a P3 split leftover — the operations group was designed to delegate to the runner/registry but that delegation was never implemented.

- **`DirectMapping` type and `mapping()` helper are copy-pasted four times.** Identical declarations appear in `vba-sync-adapter.ts:99`, `vba-modules-adapter.ts:19`, `vba-forms-adapter.ts:10`, and `vba-execution-adapter.ts:11`. The `mapping()` factory function is duplicated in three of those four files. These should live in a shared `src/adapters/vba-sync/vba-sync-types.ts` file.

- **`dysflow-vba-manager.ps1` is 3,190 lines with no extracted functions for the main action dispatch.** N2 (P3 analogue for PS) was applied to the access runner but not to the VBA manager. The VBA manager has 68 functions but the main action dispatch (starting around line 2800) is a single `if/elseif` chain with no shared abstraction.

- **The early dispatch block in `dysflow-access-runner.ps1` still uses raw `if` chains rather than calling the shared `Invoke-*` functions via a dispatch table.** Lines 1376–1416 call `Invoke-QuerySqlReadAction`, `Invoke-ListTablesAction`, etc. correctly, but `list_linked_tables`, `compare_backends`, and `list_access_files` are still absent from early dispatch (they are only handled in the main MSACCESS block, lines 1586–1647). These three actions require a full `CurrentDb()` connection even when `databasePath` is explicit.

- **`releaseCurrent!` non-null assertion bypass in `access-runner.ts:385`.** The non-null assertion was banned by P1 (`noNonNullAssertion: error`), but `let releaseCurrent!: () => void` is a definite assignment assertion (`!`), not a non-null assertion operator. Biome does not flag it. At runtime the `releaseCurrent = resolve` assignment at line 388 always executes before line 401 is reached — so this is not a bug — but it is a code smell that requires careful reasoning to verify.

- **`VbaOperationsAdapter` creates its `AccessOperationPreflightCleanupService` lazily via a dynamic `import()` inside `createDefaultPreflightCleanup`.** `src/adapters/vba-sync/vba-operations-adapter.ts:70`. This deferred import was introduced to avoid circular dependencies, but it means the import path is not statically analysable and is not covered by any test. If the module path changes (e.g., during a future restructure), the error will only surface at runtime.

- **`install-utils.ts` (227 lines) mixes four unrelated concerns**: file utilities (`fileExists`, `readJson`, `writeJson`), command runner (`runCommand`, `runCommandOutput`), path resolution (`getSystemMarkerPath`, `resolveRuntimeDir`, `getHome`, `resolveAgentConfigPaths`), and agent config removal (`removeDysflowMcpConfig`, `removeAgentConfig`). Low urgency, but explicitly identified as remaining debt.

- **`src/core/services/vba-source-comparison.ts` has three `as unknown as` double-cast expressions** (lines 101, 103, 174). The correct fix is to make `failureResult` return `OperationResult<never>` (covariant failure type). This is a TypeScript design gap, not a runtime bug.

## Opportunities for Improvement

1. **Implement `VbaOperationsAdapter.execute()` properly** [HIGH impact, LOW effort]. Replace the unconditional `TOOL_NOT_IMPLEMENTED` stub with real delegation to `operationRegistry.listRecent()` and `cleanupService.cleanup()`. Requires injecting both services into the adapter constructor. Estimated effort: 2–3 hours.

2. **Extract shared `DirectMapping` type and `mapping()` / `stringArray()` factory to a shared types file** [MEDIUM impact, LOW effort]. Create `src/adapters/vba-sync/vba-sync-types.ts`. Estimated effort: 1 hour.

3. **Migrate MCP to `@modelcontextprotocol/sdk`** [HIGH impact, HIGH effort]. P7 documented this as Option B — the correct long-term play. The hand-rolled `stdio.ts` (472 lines) handles protocol version pinning, `tools/list`, `tools/call`, progress notifications, and error wrapping manually. The official SDK handles all of this and evolves with the spec. Estimated effort: 2–3 days.

4. **Add early dispatch to `dysflow-access-runner.ps1` for `list_linked_tables`, `compare_backends`, `list_access_files`** [MEDIUM impact, MEDIUM effort]. These three read actions could bypass the `Access.Application` COM open when a `databasePath` is explicit, removing 3–8 s of MSACCESS startup overhead. Requires extracting three `Invoke-*` functions and adding them to the early dispatch block. Estimated effort: 4–6 hours + E2E verification.

5. **Add Pester unit tests for `Invoke-*` functions in `dysflow-access-runner.ps1`** [MEDIUM impact, MEDIUM effort]. The N2 refactor extracted these functions but did not add per-function Pester tests. The current `test/scripts-access-runner.test.ts` tests via the full TypeScript runner path. Estimated effort: 1 day.

6. **Apply N2-style dispatch extraction to `dysflow-vba-manager.ps1`** [MEDIUM impact, HIGH effort]. The VBA manager is 3,190 lines with a large `if/elseif` dispatch block at the bottom. Recommended as a separate SDD change. Estimated effort: 3–5 days.

7. **Split `install-utils.ts` into focused utility files** [LOW impact, LOW effort]. Separate `file-utils.ts`, `process-utils.ts`, `path-resolver.ts`, and `agent-config.ts`. Estimated effort: 2–3 hours.

8. **Fix `OperationResult` variance to eliminate `as unknown as` casts** [LOW impact, MEDIUM effort]. Change `failureResult` to return `OperationResult<never>`. Requires verifying all callers are compatible. Estimated effort: 2–4 hours.

## Probable Errors / Risks

- **`list_access_operations` and `cleanup_access_operation` via `vba-sync` MCP route silently fail.** When `MCP_TOOL_ROUTES["list_access_operations"] = { kind: "vba-sync" }` routes through `vbaSyncToolService.execute()`, it reaches `VbaOperationsAdapter.execute()` which returns `TOOL_NOT_IMPLEMENTED`. The only reason these tools currently work is that legacy alias handlers at `tools.ts:264–301` are registered before the loop at line 416. Any future re-ordering of registration — or a new consumer calling `vbaSyncToolService.execute()` directly — will produce silent failures. Reference: `src/adapters/vba-sync/vba-operations-adapter.ts:37–39`, `src/adapters/mcp/tools.ts:416–421`.

- **Checksum verification falls back to `git clone` silently on any non-checksum error.** `src/cli/commands/install/downloader.ts:183–206`: the `catch` block re-throws only if the error message contains `"Checksum mismatch"`. Any other download error (404, timeout, malformed archive) triggers a `git clone` fallback that skips checksum verification entirely. A targeted 404 on the archive URL would silently install from git, bypassing the security control. Fix: only fall back when the archive URL returns HTTP 404, not for all errors.

- **`compact_repair` is reachable twice in `dysflow-access-runner.ps1`.** An early exit at line 1449 handles `compact_repair`. If the string check fails (e.g., unexpected casing), execution falls through to the MSACCESS open, and a second handler at line 1717 calls `CompactDatabase` with the database already open in MSACCESS — which DAO explicitly warns against. Latent, not likely to trigger, but worth noting.

- **`looksLikeReadOnlySql` heuristic in `server.ts:256–278` is not a reliable SQL gate.** Access SQL supports `SELECT * FROM myActionQuery` where `myActionQuery` is a stored action query that performs writes. Such a string passes the heuristic (first token is `select`, no `INTO`) but executes writes. The real guard is `writesEnabled` — the comment correctly states this — but the heuristic creates a false sense of read-only safety. Risk is low given localhost-only binding but increases with LAN exposure or `httpToken` misconfiguration.

- **Cross-process lock uses MD5 for path hashing.** `src/core/runner/access-runner.ts:347–349`: `getCrossProcessLockPath` hashes `accessPath.toLowerCase()` with MD5. Not a security issue, but MD5 collision probability (1 in 2^64) is non-negligible on systems with hundreds of concurrent `.accdb` files. Two unrelated databases could contend on the same lock file. Idiomatic alternative: SHA-256 truncated to 16 hex chars, or a sanitized path-derived filename.

- **`spawnPowerShellProcess` in `powershell-executor.ts:24–33` spawns `taskkill` fire-and-forget on timeout.** If `taskkill /T /F /PID` itself fails (e.g., Access already exited), the spawn error is silently discarded. COM sub-processes launched by Access that outlive the `taskkill` will not be cleaned up, potentially holding `.laccdb` locks.

- **`resolveTestProceduresJson` reads `tests.vba.json` without a size limit.** `src/adapters/vba-sync/vba-execution-adapter.ts:133–135`: an arbitrarily large file will be fully buffered into memory before validation. Not exploitable in the current MCP-local model but worth noting alongside the HTTP server's body size limit.
