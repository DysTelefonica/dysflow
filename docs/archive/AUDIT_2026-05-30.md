# Dysflow — Audit Report v1.2.2 (2026-05-30)

## Summary

This audit supersedes `AUDIT_2026-05-29.md` (v0.9.20). In the v1.0.0 → v1.2.2 sprint the
project closed most of the previously open structural debt: the MCP SDK migration shipped,
the `VbaOperationsAdapter` stub was implemented, the `DirectMapping` duplication was
extracted to a shared module, `install-utils.ts` was split, the checksum-verification silent
fallback was removed, and orphaned `MSACCESS.EXE` cleanup was hardened. The architecture is in
its best shape yet.

The trade-off: stability of the **install/update path** regressed under a runtime dependency
(`@modelcontextprotocol/sdk`) and required four consecutive hotfixes (v1.0.1, v1.0.2, v1.2.1,
v1.2.2). Coverage discipline also slipped — the branch threshold was lowered from 82 % to 77 %
in the most recent commit as explicitly tracked debt. The two PowerShell scripts remain the
largest untested-by-unit surface and grew rather than shrank.

## Strengths

- **Architecture is clean and enforced.** Strict one-way dependency (`CLI/adapters → core/services → runner → PowerShell`), `src/core` returns typed `OperationResult`, adapters only translate at the boundary. There is an `test/architecture/` suite gating this.
- **Protocol mechanics are no longer hand-rolled.** The MCP stdio adapter now sits on `@modelcontextprotocol/sdk` v1.29.0. Custom behaviors (error absorption, path sanitization, hidden tools, 1 MiB size guard) survive as focused wrapper modules instead of a 470-line bespoke runtime.
- **Safety model is genuinely strong.** Operation registry as source of truth, cleanup gated behind seven validated refusal codes (`CLEANUP_*`), VBA `allowedProcedures` allowlist enforced on all three entry points, optional HTTP Bearer auth, 1 MiB request body caps on both MCP and HTTP, SHA-256 verification on `update`.
- **Adapter sub-modules are well-sized.** The `vba-sync` adapters and the `install/` sub-modules are all comfortably under the 260-line ceiling, with shared types factored out.
- **Process-lifecycle correctness was a real focus.** v1.2.2 addresses COM release ordering, PID re-resolution after COM singleton reuse, deterministic termination waits, and per-call zombie checks in the E2E suite.

---

## Weaknesses

- **Branch coverage threshold lowered 82 % → 77 %** (`vitest.config.ts:28`, latest commit). This is acknowledged debt ("while coverage debt is tracked"), but it is a regression in a quality gate — the gate now passes by lowering the bar, not by adding tests.
- **Install/update path is fragile.** Four consecutive releases (v1.0.1, v1.0.2, v1.2.1, v1.2.2) were hotfixes for the same area: missing prod deps, `--prefer-offline` ETARGET, npm 11.7.0 null-pointer crash, hanging network calls with no timeout. Introducing the first true runtime dependency exposed an under-tested installer.
- **`pnpm` is now a hard runtime prerequisite for `update`.** `dysflow update` builds the release in a temp workspace and shells out to `git`, `pnpm install`, `pnpm build`, and `tar`. That is a heavy, environment-sensitive chain for an end-user "update" command.
- **The two PowerShell scripts are the dominant untested-by-unit surface and are growing.** `dysflow-vba-manager.ps1` is **3,245 lines** (was 3,190) and `dysflow-access-runner.ps1` is **1,881 lines** (was 1,762). Pester test files exist but the core action-dispatch logic — where the real Access automation lives — is still a large `if/elseif` chain.
- **Documentation drift.** `README.md:17` still says "Current version: v1.0.0" and "717 tests" while the project is at v1.2.2 with ≈670 collected test cases. The prior `AUDIT_2026-05-29.md` now describes fixed issues as open, which actively misleads.
- **Bleeding-edge dev toolchain.** `package.json` pins `typescript ^6.0.0`, `vite ^6`, `vitest ^4`, `@types/node ^22` while `engines.node` is `>=20`. Installed `node_modules` still shows older transitive `vitest@1.6.1`/`@3.2.4` and `@types/node@20`/`@22` — a sign of version churn worth reconciling.

---

## Opportunities for Improvement

1. **Pay down the coverage debt and restore branches to 82 %** [HIGH impact, MEDIUM effort]. The threshold drop should be temporary; add the missing branch tests (most gaps are in the install/download and PS-runner glue) and raise the gate back.
2. **Add an installer/update smoke test in CI** [HIGH impact, MEDIUM effort]. Four production hotfixes in one area means the path is untested end-to-end. A matrix run (npm vs pnpm, clean profile) would have caught all four.
3. **Decompose `dysflow-vba-manager.ps1`** [MEDIUM impact, HIGH effort]. Extract the action dispatch into a table-driven map of `Invoke-*` functions, mirroring what was started in the access runner. Best as its own SDD change.
4. **Reconcile and pin the dev toolchain** [MEDIUM impact, LOW effort]. Decide whether TS 6 / vitest 4 / vite 6 are intentional, regenerate the lockfile, and remove stale transitive versions.
5. **Refresh user-facing docs to v1.2.2** [LOW impact, LOW effort]. Fix the README version/test-count banner; archive or mark the two stale AUDIT files as superseded.
6. **Deepen Pester coverage to per-function level** [MEDIUM impact, MEDIUM effort]. The test files exist; assert behavior of individual `Invoke-*` / `Find-AccessPidByDatabase` / `Stop-AccessPidAndWait` functions, not just full-runner paths.

## Probable Errors / Risks (current)

- **`update` depends on a heavy, environment-sensitive toolchain.** `dysflow update` needs `git` + `pnpm` + `tar` + network and builds from source in a temp dir. On a machine without `pnpm`, or with a different npm/pnpm version, update fails. The recent npm-11 crash is the canonical example — the command is doing a full source build where a packaged artifact would be more robust.
- **Coverage gate at 77 % can hide regressions.** With the branch bar lowered, new untested branches can land green. The risk is concentrated exactly where the recent bugs were (install/download, PID resolution).
- **`looksLikeReadOnlySql` heuristic remains an advisory, not a guard** (`server.ts`). `SELECT * FROM <actionQuery>` can perform writes yet pass the first-token check. The real guard is `writesEnabled` (localhost-bound by default), so risk is low — but the heuristic still invites a false sense of read-only safety on LAN exposure.
- **Cross-process lock still hashes the path with MD5** (`access-runner.ts`). Not a security issue; collision probability is low but non-zero across many concurrent `.accdb` files. SHA-256 truncated would be the idiomatic fix.
- **Stale `node_modules` vs declared deps.** The installed tree shows older vitest/@types versions than `package.json` declares. A `pnpm install` mismatch on a fresh checkout could surface as confusing build/test failures.
