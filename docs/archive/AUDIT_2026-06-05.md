# Audit — dysflow v1.2.18 (2026-06-05)

Fresh read of ~70 `src` files plus the test suite. Strengths, weaknesses, improvement
opportunities, and technical debt. Findings cite `file:line` as evidence.

## 1. Strengths

**Architecture — the hexagonal boundary is real, not decorative.**
- `src/core` imports nothing from `src/adapters`, enforced by `test/architecture/core-boundary.test.ts`.
- `VbaSyncPort` uses `string` instead of adapter types, with rationale (`contracts/index.ts:47-49`).
- `OperationResult<T>` is a discriminated union forcing an `ok` check before reading `data` (`contracts/index.ts:23-37`).

**Security — layered defense in depth.**
- Secret sanitization layered and ordered: literal → DAO connect-strings → path strip (`sanitize-error.ts:7-12`), also applied before persisting to the registry (`access-runner.ts:222-229`).
- `timingSafeEqual` for the HTTP Bearer token (`server.ts:160`).
- `SizeLimitTransform`: 1 MiB per-line guard with correct `-32700` frames (`stdio-size-guard.ts:21`); double body guard in HTTP (`server.ts:343-364`).

**MCP — single source of truth.**
- `TOOL_PARITY_REGISTRY` is the sole source for stub/hidden state (`tool-parity-registry.ts:98-109`); `Record<DysflowMcpToolName, McpToolRoute>` makes a missing tool a compile error.

**Concurrency over a single-threaded COM target.**
- Dual lock (in-process + filesystem) with anti-stale heartbeat (`access-runner.ts:118,353,390`); atomic registry writes via temp-file + rename (`access-operation-registry.ts:274-289`).

**Testing — the port layer is exemplary.**
- `test/core/services/core-services.test.ts`, `access-runner.test.ts`, `access-operation-preflight.test.ts` exercise real logic injecting ONLY the I/O adapters. Textbook application of the repo's philosophy.

## 2. Weaknesses

**High:**
- `VbaFormService` fakes DI: receives `executor`, `resolveExecutionTarget`, `validateStrictContext` typed as `unknown` and never uses them — calls `readdir`/`writeFile` directly (`vba-form-service.ts:12-31`). Dead code disguised as injection; impossible to unit-test in isolation.
- `dispatch.ts` is a 513-line god file mixing route table, alias handlers, factory, and registration.
- Domain logic leaked into the adapter: SQL keyword extraction reimplemented in `dispatch.ts:106-115` on top of core's `looksLikeReadOnlySql` → two keyword lists that can drift.

**Medium:**
- `writesDisabled()` / `invalidInput()` duplicated verbatim in `tools.ts:44-54` and `dispatch.ts:33-44`.
- `resolveExecutionTarget` (75 lines of config/path resolution) lives in the adapter when it is use-case logic (`vba-sync-adapter.ts:362-436`).
- `vba-sync-adapter.ts:601-607` re-exports **core** symbols from an adapter → invites coupling to the wrong layer.
- `config` baked into service constructors when the runner already accepts config per-call — statefulness without benefit.

## 3. Improvement opportunities (concrete refactors)

1. Unify path matching. `pathMatchesAccessPath` and `normalizePathForMatching` are duplicated in `access-operation-cleanup.ts:250-268` and `access-operation-preflight.ts:326-355`, **and already diverged** (preflight strips slashes, cleanup does not). Extract to `src/core/utils/path-utils.ts` and fix the inconsistency. See D1 — it is a latent bug.
2. Split `dispatch.ts` into route-table / alias-tools / dispatch-factory (each <150 lines).
3. Move `rejectWriteSqlInReadMode` entirely into core; leave the adapter to format the message only.
4. Inject `VbaFormService` into `VbaFormsAdapter` instead of instantiating it (`vba-forms-adapter.ts:43`).
5. `TOOL_PARITY_REGISTRY` → `Map`. Today it does `Array.find()` O(n) on every dispatch over 45 entries (`tool-parity-registry.ts:112-118`).
6. Shared `parseNamedArgs(specs, args)`. Five hand-rolled arg parsers with the same `args[i+1]` lookahead and the same `startsWith("--")` guard copied (updater/uninstall/setup/serve/relink-directory).
7. Replace text-over-`.ps1` tests with behavior tests via Pester (`scripts/tests/`) or integration that observes the output JSON.

## 4. Technical debt

**High:**
- **D1 — Latent bug from path-normalization divergence.** Cleanup does NOT strip leading/trailing slashes (`cleanup.ts:264-268`); preflight does (`preflight.ts:326-331`). A path with `./` matches in preflight but not in cleanup → preflight passes but a manual cleanup of the same operation fails.
- **D2 — Brittle PowerShell JSON parsing.** `indexOf("{")`/`lastIndexOf("}")` to slice JSON from stdout (`vba-sync-adapter.ts:526-547`, also `access-runner.ts:506-525`). Any `{` in prior diagnostic output corrupts the slice, and the fallback masks real parse failures. Real fix: separate the JSON from the stream (the `DYSFLOW_PROGRESS` marker system already exists — the stdout heuristic undermines it).
- **D3 — `withTimeout` does not kill the child process.** Timeout rejects the promise but the process stays alive holding locks → `EBUSY` on Windows on subsequent ops (`command-runner.ts:17-33`).
- **D4 — `applyIntegrationSelection` removes agent configs not installed by dysflow** without checking an "installed by dysflow" marker (`install.ts:71-77`).

**Medium:**
- `timeoutMs` and `processTimeoutMs` always set equal (`dysflow-config.ts:207,295`) — distinction never implemented or one field is dead.
- `AccessQueryRequest.action`: 25-variant union with no runtime validation of a known value before serializing to the `.ps1` → unknown value reaches PowerShell opaquely (`contracts/index.ts:108-133`).
- `test/docs/architecture-doc.test.ts:11` asserts a machine-absolute path (`C:\\Proyectos\\workflow\\skills\\dysflow`) → fails on any other machine.
- Integration tests for critical paths (apply/backup/relink) gated on undocumented env vars → 3 of 4 silently skip (`access-relink-apply.test.ts:48-53`).
- E2E gated on `.accdb` files not in source control → pass as skipped if absent, no "fixture missing" guard (`access-fixture.e2e.test.ts:18-22`).

**Low:**
- `MAX_PACKAGE_ROOT_DEPTH=12` and silent fallback to `process.cwd()` / `"0.0.0"` (`package-info.ts:6`, `package-root.ts:36`).
- Error detection by string prefix `"RUNNER_LOCK_TIMEOUT:"` instead of an error class (`access-runner.ts:292`).
- `RUNTIME_MARKER_VERSION="1"` never incremented; `parseRuntimeMarker` misreads future formats (`runtime-dir.ts:5,18-28`).
- `SizeLimitTransform` mixes `.length` (chars) with `byteLength` (bytes) → the early-drop optimization misses multi-byte UTF-8 (`stdio-size-guard.ts:51,73`).

## Architect's read

The core is solid: the hexagonal boundary is tested and respected, and the port layer has model
tests. That is the hard part, and it is already built.

The fat is in two concrete places:

1. **The CLI/install/uninstall edge** is where the real danger lives — destructive filesystem
   operations on the user's machine without guards (C1, D4) and silent data loss (C3). This is not
   "someday" debt; it is where a user loses data.
2. **The text-over-PowerShell tests** (C4) are a maintenance trap that contradicts the repo's own
   written philosophy. Every behavior-preserving `.ps1` refactor turns CI red. The test is the
   defect, as `testing-philosophy.md` states.

One-week priority: **C1, C3, and D1** (the two data footguns + the path bug that already exists).
Everything else is quality improvement, not risk.
