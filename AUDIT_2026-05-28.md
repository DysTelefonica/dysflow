## тмЬ Pending: N1 тАФ Legacy Tool Dispatch Cascade (TypeScript)

### Problem

`appendLegacyCompatibilityTools` in `src/adapters/mcp/tools.ts` has **5 places to keep in sync**:

1. `LEGACY_VBA_SYNC_TOOL_NAMES` (array) тАФ tools dispatched to `legacyToolService`
2. `LEGACY_QUERY_SLICE_TOOL_NAMES` (array) тАФ query read tools
3. `LEGACY_QUERY_MAINTENANCE_SLICE_TOOL_NAMES` (array) тАФ query maintenance tools
4. `LEGACY_WRITE_FIXTURE_SLICE_TOOL_NAMES` (array) тАФ fixture write tools
5. `getLegacyParityToolDefinition` from `legacy-tool-inventory.ts` тАФ parallel parity registry

Drift is already present: `tools.ts` local arrays and `legacy-tool-inventory.ts` are separate and must be manually kept consistent. Adding a new legacy tool requires touching 2-4 of these 5 locations.

### Target State

Replace all 4 `LEGACY_*_TOOL_NAMES` arrays + the `createLegacyDispatchTool` cascading `if` chain with a single table:

```typescript
type LegacyToolEntry = {
  schema: JsonObjectSchema;
  queryMode?: "read" | "write";    // undefined = VBA sync / special dispatch
  requestBuilder?: (input: unknown) => AccessQueryRequest;
};

const LEGACY_TOOL_TABLE: Record<string, LegacyToolEntry> = {
  query_sql:    { schema: legacySchemaFor("query_sql"),    queryMode: "read",  requestBuilder: ... },
  exec_sql:     { schema: legacySchemaFor("exec_sql"),     queryMode: "write", requestBuilder: ... },
  list_tables:  { schema: legacySchemaFor("list_tables"),  queryMode: "read",  requestBuilder: ... },
  // ... all 43 legacy tools
};
```

Then `appendLegacyCompatibilityTools` becomes:

```typescript
for (const [name, entry] of Object.entries(LEGACY_TOOL_TABLE)) {
  add(createLegacyTool(name, entry));
}
```

### TDD approach

1. The existing 49 test files (645 tests) are the RED тЖТ GREEN contract тАФ all must pass after the refactor.
2. Add a regression test that `Object.keys(LEGACY_TOOL_TABLE).length === 43` and each key is in `LEGACY_TOOL_SCHEMAS`.
3. Add a test that `createDysflowMcpTools` still advertises exactly 48 tools.

### Key files

- `src/adapters/mcp/tools.ts` тАФ `appendLegacyCompatibilityTools`, 4 arrays, `createLegacyDispatchTool`
- `src/adapters/mcp/legacy-tool-inventory.ts` тАФ `getLegacyParityToolDefinition`, parity registry
- `test/adapters/mcp/legacy-parity.test.ts` тАФ parity contract tests
- `test/adapters/mcp/tools.test.ts` тАФ main tools test suite

### Risk

MEDIUM. This is a pure TypeScript refactor with full test coverage. The behavioral output (tool schemas, request forwarding, write guards) must remain identical. No PS or E2E tests required тАФ unit tests are the acceptance criterion.

## тмЬ Pending: N2 тАФ PS Dual Read-Dispatch Unification

### Problem

`dysflow-access-runner.ps1` has **two parallel dispatch blocks** for the same read actions:

**Early dispatch** (lines ~1292-1383): triggers when `$earlyTargetPath` (explicit `databasePath`/`backendPath`) is provided. Opens the target DB directly via DAO `OpenDatabase`. Handles: `query_sql`, `list_tables`, `get_schema`, `count_rows`, `distinct_values`, `get_relationships`.

**Main block** (lines ~1420-1700): runs after MSACCESS opens and `CurrentDb()` is called. Handles the same 6 PLUS `compare_backends`, `list_access_files`, `list_links`, `get_relationships`, and more.

Consequence: adding a read action requires updating both blocks. They have already drifted тАФ early dispatch handles 6 actions, main block handles 12+.

### Target State

Extract each shared read action into a named function that accepts a `$Database` object:

```powershell
function Invoke-QuerySql      { param($Database, $Payload) ... }
function Invoke-ListTables    { param($Database, $Payload) ... }
function Invoke-GetSchema     { param($Database, $Payload) ... }
function Invoke-CountRows     { param($Database, $Payload) ... }
function Invoke-DistinctValues { param($Database, $Payload) ... }
function Invoke-GetRelationships { param($Database, $Payload) ... }
```

Both dispatch blocks call the same function тАФ only the `$Database` argument differs (early = DAO-opened `OpenDatabase`; main = MSACCESS `CurrentDb()`).

### TDD approach

1. Build + install to `test-runtime` after each function extraction.
2. Run `test-sql-guard.mjs` E2E after each extraction (fast, no fixtures needed).
3. Run the full `mcp-e2e.mjs` E2E once at the end (requires .accdb fixtures).
4. The Pester test suite (`pnpm run test:ps`) must stay green throughout.

### Risk

HIGH. PS refactors have no TypeScript unit test coverage. Strategy:
- Extract one function at a time (not all at once).
- Each extraction = one commit with E2E verification.
- Keep the early dispatch block intact until all 6 shared functions are extracted and verified.
- Then collapse both blocks to call the same functions.

### Key file

- `scripts/dysflow-access-runner.ps1` тАФ ~1720 lines, early dispatch ~1292-1383, main block ~1420-1700
