## ⬜ Pending Work

### I — Schema validator: `minLength` + `pattern` for critical fields

**Why**: `validator.ts` only validates `type`, `enum`, `required`, `additionalProperties`, `properties`, `items`. Fields like `procedureName`, `tableName`, `sql` accept empty strings — the error surfaces late in the PS runner with a cryptic message instead of an early, clear MCP response.

**Where**: `src/adapters/mcp/validator.ts` + `src/adapters/mcp/schemas.ts`

**TDD approach**:
1. RED: test that `dysflow_vba_execute` with `procedureName: ""` returns `MCP_INPUT_INVALID` before reaching the runner
2. Add `minLength` support to `validateJsonSchemaProperty` in `validator.ts`
3. Add `minLength: 1` to `procedureName`, `tableName`, `sql` fields in `schemas.ts`
4. GREEN

**Scope**: additive only — existing callers passing valid strings are unaffected.

---

### J — `Restore-StartupFeatures` sentinel on hard-kill

**Why**: When `taskkill /T /F` kills the PS process tree, the PS `finally` block doesn't run. The `.accdb` is left with AutoExec/StartupForm permanently disabled until the next dysflow run explicitly restores them.

**Where**: `scripts/dysflow-access-runner.ps1`

**Approach**: Write a `{accessPath}.dysflow-startup-restore.json` sentinel BEFORE `Disable-StartupFeatures`. At the start of the NEXT run against the same `.accdb`, check for this sentinel and restore startup features if present. Clean up the sentinel on success.

**TDD approach**: Unit test at PS level is impractical (COM dependency). E2E acceptance: after a simulated hard-kill, the next run succeeds without leaving the db crippled.

---

### K — Filesystem cross-process lock for `.accdb`

**Why**: `accessExecutionLocks` in `access-runner.ts:112` is a module-level Map — invisible to other processes. Two concurrent `dysflow` CLI instances against the same `.accdb` will both proceed and corrupt each other's DAO session.

**Where**: `src/core/runner/access-runner.ts`

**Approach**: Before spawning PS, create `${accessDbPath}.dysflow.lock` dir using the same mkdir-as-EEXIST pattern as `FileAccessOperationRegistry`. Release after PS completes. Stale detection: if mtime > `CROSS_PROCESS_LOCK_STALE_MS` (30s — much shorter than registry since this lock is held only during PS execution), steal it.

**TDD approach**: Test that two concurrent `run()` calls against the same path serialize (second waits until first completes). Lock-acquisition timeout → fail with a clear `RUNNER_LOCK_TIMEOUT` error.

---

### L — `compact_repair`: skip frontend open when targeting backend only

**Why**: `compact_repair` with `databasePath = backendPath` still launches MSACCESS against the frontend (needed for Access COM automation), then compacts a different file. This is unnecessary overhead and holds the frontend lock.

**Where**: `scripts/dysflow-access-runner.ps1` — compact_repair action

**Approach**: When `$Payload.databasePath` differs from `$AccessDbPath` (i.e., it's a backend-only compaction), use pure DAO (`JRO.JetEngine` or `DBEngine.CompactDatabase`) instead of the Access COM object. No MSACCESS window needed.

**Caveat**: `JRO` may not be available on all Access installations. Needs version check.

---

### M — `Update-LinkTables` password warning in schema

**Why**: When `dysflow` links tables with a backend password, Access stores the password inside the `.accdb` file's TableDef Connect string. This is by-design Access behavior, but users may not realize their password is persisted in the database file.

**Where**: `src/adapters/mcp/schemas.ts` — `link_tables` and `relink_tables` tool descriptions

**Approach**: Add a note to the tool description: "When backendPassword is provided, Access stores the credential in the linked table's Connect string inside the database file."

**Scope**: Documentation only — no code change.
